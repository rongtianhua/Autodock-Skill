# Ollama 模型管理经验备忘

## Ollama 双进程架构

Ollama 运行时会创建两个进程：

| 进程 | 角色 | 说明 |
|------|------|------|
| `ollama serve` | 主服务/网关 | Ollama 的常驻服务进程 |
| `ollama runner` | 模型运行时 | 实际加载和运行模型的进程 |

## 当前 Mac mini 已注册模型

| 模型 | 大小 | 用途 | 是否占用本地 |
|------|------|------|------------|
| `bge-m3:567m` / `:latest` | 1.2 GB | MemOS 记忆系统向量嵌入 | ✅ 本地 |
| `kimi-k2.5:cloud` | 未知 | 云端模型（纯调用 API） | ❌ 不占本地 |

## 模型删除决策准则

**谨慎保留**（不易删）：
- 标称 `cloud` / `云端` 的模型 → 只是注册信息，实际调用远程 API
- `bge-m3` → MemOS 记忆检索依赖的向量嵌入模型

**可安全删除**：
- 明确是本地运行且不再使用的模型（如 Qwen3.5、Qwen3-VL-Embedding）

## 常用命令

```bash
# 列出所有模型
ollama list

# 删除模型
ollama rm 模型名称

# 重新拉取（云端模型也会重新注册）
ollama pull kimi-k2.5:cloud
```

## 内存占用

Ollama（serve + runner）日常占用约 **1.5 GB**，关闭后可释放。

如需释放内存但不停止服务：确保没有 runner 进程在跑（加载模型后才会出现）。
