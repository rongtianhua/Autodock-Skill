## 2026-05-05 — 结构数据源修复

### Bug #1：ZINC22 URL 格式损坏（全部 404）✅ 已修复

**根因**：两层错误
1. 文件后缀：代码用 `.txt.gz`，但 ZINC22 服务器只提供 `.smi.gz`（`.txt.gz` 根本不存在）
2. MW branch 目录名：`mw_b=0` 生成 `H05M000/` 但代码只写 `"000"` 而非 `"M000"`

**修复** `_autodock.py`：
```python
# 修复前（旧）：
mw_str = f"{mw_b:03d}"      # "000" → 错误，应为 "M000"
url = f"...H05/H05M000/H05M000-N.g.txt.gz"  # ❌ 404

# 修复后（新）：
mw_str = f"M{mw_b:03d}"     # "M000" ✅
url = f"...H05/H05M000/H05M000-N.g.smi.gz"  # ✅ 200 OK
```

同时 `_zinc_tranche_url()` 函数也更新为只返回 LogP branch（删除了有问题的 MW bucket 逻辑）。

**验证**：H05P030-N.g.smi.gz 实地探测 ✅（21 lines）；H04P020-N.g.smi.gz ✅（1 line）

**注**：MW branch 仍只有 H05M000/H05M100 两个目录存在（服务器物理限制），对于高 MW 化合物（>200 Da）MW branch 会一直 404；这是 ZINC22 服务器端结构问题，无法通过代码修复，但 LogP branch 提供完整覆盖。

### Bug #2：AlphaFold ATOM 检测阈值偏低 ✅ 已修复

**根因**：用字符串包含 `'ATOM' in content` 会错误匹配 `ATOM0001` 等行尾，且对含非 ASCII 字符的 AlphaFold 文件可能漏检。

**修复** `_structure_fetch.py`：
```python
# 修复前：
if 'ATOM' not in content:
    raise ValueError(f"AlphaFold structure not found: {uniprot_id}")

# 修复后（2026-05-05）：
if not re.search(r'^ATOM ', content, re.MULTILINE):
    raise ValueError(f"AlphaFold structure has no ATOM records: {uniprot_id}")
```

**验证**：正则可正确识别 `ATOM      1    N ...` 行，拒绝 `ATOM0001` 类非标准格式。

### Bug #3：PubChem SDF 空文件 Fallback ✅ 已修复

**根因**：Nirmatrelvir 等大分子/多手性中心的化合物，PubChem PUG REST 返回的 SDF 仅 ~4.7KB（坐标截断）。

**修复** `_structure_fetch.py` `fetch_molecule_pubchem()`：
```python
# 下载后验证 SDF 有效性（V2000 header + size > 5KB）
# 若无效，自动用 RDKit ETKDGv3 生成 3D 结构
if sdf_valid:
    ...缓存正常 SDF
else:
    # Fallback：SMILES → RDKit ETKDGv3 → 完整 3D SDF
    mol = Chem.MolFromSmiles(smiles)
    mol = Chem.AddHs(mol, addCoords=True)
    AllChem.EmbedMolecule(mol, params_etkdg)
    AllChem.MMFFOptimizeMolecule(mol)
    Chem.SDWriter(output_sdf).write(mol)
    # 同样缓存到 ~/.openclaw/structures_cache/
```

**验证**：Nirmatrelvir（35 个重原子）ETKDGv3 生成 5664 字节有效 SDF，含 V2000 + 67 个 3D 坐标 ✅
