### 失败原因
```
message failed: Action send requires a target.
raw_params={"action":"send","message":"🦾 老板提醒..."}
```

**根本原因**：
1. 创建 cron 定时任务时漏了 `--target` 参数
2. cron 在独立会话（isolated）里运行，没有携带用户上下文，不知道该发给谁
3. 只设了 `--channel feishu` 但没有指定具体用户

### 修复方法
- 补发了提醒
- 重建定时任务，添加正确的 `--target` 参数（用户的飞书 open_id）

### 教训总结
1. **创建飞书定时任务必须指定 target**：`openclaw cron add --channel feishu --target <user_open_id> --at "09:50" --message "..."`
2. cron 在 isolated 模式下运行，无法继承当前会话的用户上下文
3. 创建后要验证任务是否成功建立

## a-stock-trader-pro 调试成功经验总结 (21:06)

### 本次调试背景
- 之前多次调试效果不理想，数据源不稳定
- 本次调试产出了完整的 v2.4.0 升级

### 成功关键经验

#### 1. 先写测试脚本，再改代码
- 发现问题后，先写独立的 test_xxx.py 验证假设
- 确认原始数据正确后，再修改业务代码
- **关键**：不依赖 print 猜数据，直接打印原始 parts 数组索引对照

#### 2. 新浪五档盘口教训
- **错误**：用了和腾讯一样的索引 [9,11,13,15,17] 
- **正确**：新浪和腾讯字段完全不同，parts[7]=当前价，被错当 bid5
- **经验**：同一数据源的多个字段，必须先打印全部原始 parts 对比，验证每个索引的含义
- **验证方法**：bid5(腾讯)=1446.78，新浪[19]=1446.78 ✅ 才确认正确

#### 3. 东财 API 替代 Playwright
- **问题**：东财触发了人机验证码弹窗，Playwright 渲染整个页面失效
- **解决思路**：
  1. 先用 curl/requests 测试哪些域名可访问（push2.eastmoney.com 不通，push2delay 可通）
  2. 用 `curl -w "%{http_code}"` 批量探测可用域名
  3. 找到 API 后，Python requests 直连测试（发现 akshare 的代理影响整个 venv）
  4. **经验**：先在系统 shell 层面验证网络，再进 Python 环境

#### 4. 字段研究方法
- 用全量字段请求，一次拿到所有 f 参数
- 对比茅台/平安/宁德多只股票的同一字段，找规律
- 计算验证：f173% × 成交额 ≈ f184 亿元，交叉验证字段单位
