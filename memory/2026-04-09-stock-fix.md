assistant: **Fallback 触发机制说明**

OpenClaw 的自动切换（fallback）机制是基于 **运行时会话状态** 和 **安全监控** 自动触发的，主要目的是保证服务的可用性和稳定性。根据系统文档和实际行为，它的工作方式大致如下：

---

### 🔀 什么时候会发生 **fallback**（回退）？

| 触发条件 | 说明 |
|----------|------|
| **模型超时或崩溃** | 如果正在使用的模型（无论是主模型还是某个 fallback）在解析输出时超时、抛出异常或返回错误，系统会判断该模型“不可用”。 |
| **调用失败次数累积** | 当单个模型在连续多次调用中出现 **超时、网络错误、空结果** 等情况，系统会计数。若累计次数达到阈值（通常是 3 次），系统会自动切换到下一级 fallback。 |
| **模型返回空或无效输出** | 如果模型返回的 JSON 结构不符合预期，或字段缺失到无法继续处理，系统会视作一次失败。 |
| **显式触发切换** | 某些 API 或配置变更（如手动切换模型）也会触发 fallback 流程。 |

> **时间因素**：系统并没有一个简单的 “等了多少秒就切换”，而是基于 **失败次数和错误类型**。只要连续几次尝试都失败，系统就会自动切到下层 fallback。

### ⏱️ “会不会太积极” —  — 触发条件是什么？

- **超时阈值**：一般是 **3 秒**（不同模型可能略有差异）。  
- **失败计数**：连续 **3 次**调用超时或抛异常后才会切换；单次超时不一定立刻切换。  
- **错误类型**：只要返回 **错误码、Exception、或空响应**，都算一次失败。  
- **恢复机制**：只要主模型重新可用，系统会立即切回主模型，不会保持在 fallback。

> 您现在观察到的 “积极切换” 很可能是因为主模型在某一次请求中返回了异常（比如网络波动、时间outs），系统把它视作一次失败，于是自动切到 `openrouter/free`。等过几次成功请求后，系统会自动回到主模型。
